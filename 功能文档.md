## 1. 系统概述

**智能商品详情与营销文案生成系统** 是一套基于多模态大模型的电商内容管理平台。系统面向电商运营人员，利用 AI 自动生成商品详情描述、多平台营销文案和商品详情图片，大幅提升电商内容生产效率。

### 核心能力

| 能力 | 说明 | 使用模型 |
|------|------|---------|
| 商品描述生成 | 基于商品图片自动生成短标题、卖点、详情描述 | 通义千问 Qwen3-VL-Plus |
| 多平台营销文案 | 生成适配小红书/淘宝/抖音的平台化文案 | 通义千问 Qwen3-VL-Plus |
| 商品详情图生成 | 根据商品信息生成电商详情模块图 | DashScope Wan 2.7 |
| 一键流水线 | 顺序执行描述→文案→图片三步生成 | 以上三者的组合 |

### 用户角色

| 角色 | 权限范围 |
|------|---------|
| **普通用户 (USER)** | 管理自己的商品、使用 AI 生成功能、导出文案 |
| **管理员 (ADMIN)** | 管理所有商品、审核用户、设置配额/权限、发布公告 |

## 3. 用户认证与授权

### 3.1 功能概述

系统采用 **JWT 无状态认证**，所有 API 请求通过 `Authorization: Bearer <token>` 头携带身份信息。前端将 token 存储在 `localStorage` 中。

### 3.2 登录

**路由：** `/login`

**功能点：**

- **普通用户登录**
  - 输入用户名 + 密码，点击"登录"
  - 后端验证密码（BCrypt），检查账号状态（PENDING 不可登录、DISABLED 不可登录）
  - 成功后返回 JWT token（24小时有效）和用户信息（id、username、role、nickname、avatar、quotaTotal、quotaUsed）
  - 前端存入 Pinia store 和 localStorage，跳转至 `/dashboard`

- **管理员登录**
  - 切换"管理员登录"标签，输入用户名 + 密码
  - 请求中额外携带 `adminLogin: true`
  - 后端额外校验 role 必须为 ADMIN
  - 成功后跳转至 `/admin/dashboard`

**API：** `POST /api/auth/login`

```json
// Request
{ "username": "admin", "password": "admin123", "adminLogin": true }

// Response
{
  "success": true,
  "data": {
    "token": "eyJhbGciOi...",
    "userId": 1,
    "username": "admin",
    "role": "ADMIN",
    "nickname": "管理员",
    "avatar": null,
    "quotaTotal": -1,
    "quotaUsed": 0
  }
}
```

### 3.3 注册

**功能点：**

- 在登录页面点击"立即注册"
- 弹窗输入用户名、密码、昵称
- 注册后默认状态：角色 `USER`、状态 `PENDING`（待审核）、AI 配额 20 次
- **系统注册上限：50 人**

**API：** `POST /api/auth/register`

### 3.4 修改密码

**路由：** `/profile`（个人中心）

**功能点：**

- 已登录用户在个人中心修改密码
- 需要验证旧密码正确性
- 新密码最低 6 位

**API：** `PUT /api/auth/password`

### 3.5 路由守卫

**前端实现（`router/index.ts`）：**

- 非公开路由需要 token，否则跳转 `/login`
- 管理员路由（`meta.admin = true`）需要 `role === 'ADMIN'`，否则跳转 `/dashboard`
- 管理员访问 `/dashboard` 自动重定向到 `/admin/dashboard`

### 3.6 后端安全配置

- 公开路径（无需认证）：`/auth/login`、`/auth/register`、`/files/**`、Swagger 文档路径
- `/admin/**` 路径限制 `ROLE_ADMIN`
- 其他所有请求需认证
- CSRF 禁用（无状态 API）
- CORS 配置允许 `http://localhost:*` 和 `http://127.0.0.1:*`

## 4. 商品管理

### 4.1 功能概述

商品是系统的核心实体。每个用户（普通用户）管理自己的商品，管理员可管理所有商品。商品支持完整的生命周期：草稿 → 上架 → 下架。

### 4.2 商品列表

**路由：** `/products`（普通用户）、`/admin/products`（管理员）

**功能点：**

- **普通用户视图：** 卡片网格展示自己的商品，响应式布局（xs=1列、sm=2列、md=3列、lg=4列），每张卡片显示主图、名称、价格、状态标签
- **管理员视图：** 表格视图，支持批量操作（批量上架/下架/删除），带复选框多选
- **通用功能：**
  - 关键词搜索（匹配商品名称和短标题）
  - 分类筛选
  - 价格区间筛选（最低价/最高价）
  - 状态筛选（草稿/上架/下架）
  - 分页（普通用户每页 12 条，管理员每页 20 条）

**API：** `GET /api/products?page=&size=&keyword=&categoryId=&minPrice=&maxPrice=&status=`

### 4.3 新建商品

**功能点：**

- 在商品列表页点击"新建商品"
- 创建默认商品：名称"新商品"、价格 99、库存 100、状态 DRAFT
- 自动跳转到编辑页面进行详细编辑

**API：** `POST /api/products`

### 4.4 编辑商品

**路由：** `/products/:id/edit`

这是系统功能最集中的页面，包含基础信息编辑、图片管理、和全部 AI 功能。

#### 4.4.1 基础信息编辑

| 字段 | 类型 | 说明 |
|------|------|------|
| 商品名称 | 文本 | 最长 128 字符 |
| 价格 | 数字 | 保留两位小数 |
| 分类 | 下拉选择 | 从分类列表加载 |
| 库存 | 整数 | — |
| 状态 | 下拉选择 | 草稿 / 上架 / 下架 |
| 关键词 | 文本 | 用户补充，辅助 AI 生成 |
| 短标题 (AI) | 文本 | 最长 15 中文字符，AI 自动生成 |
| 核心卖点 (AI) | 动态列表 | 3-5 条，AI 自动生成，可手动增删改 |
| 详情描述 (AI) | 文本域 | 150-300 字符，AI 自动生成 |

**API：** `PUT /api/products/:id`

#### 4.4.2 批量操作（管理员）

| 操作 | 说明 |
|------|------|
| 批量上架 | 将选中商品的 status 改为 ON_SHELF |
| 批量下架 | 将选中商品的 status 改为 OFF_SHELF |
| 批量删除 | 删除选中商品及其所有图片（文件+记录） |

**API：** `PATCH /api/products/batch-status`、`POST /api/products/batch-delete`

### 4.5 删除商品

**功能点：**

- 删除商品时级联删除：
  1. 从磁盘删除所有关联图片文件
  2. 从数据库删除所有 `product_image` 记录
  3. 从数据库删除 `product` 记录
- 权限检查：只有商品所有者或管理员可删除

**API：** `DELETE /api/products/:id`

## 6. 商品图片管理

### 6.1 功能概述

每个商品最多上传 **10 张**图片。支持单张上传、批量上传、拖拽排序、主图设置。

### 6.2 图片上传

**功能点：**

- **支持格式：** JPEG、PNG、WebP
- **单张最大：** 5 MB
- **数量上限：** 每个商品 10 张
- **上传方式：** 单张上传 或 批量上传（一次选多张）
- **存储路径：** `{STORAGE_ROOT}/uploads/{productId}/{uuid}_{filename}`
- **主图规则：** 第一张上传的图片自动设为主图
- 前端使用 Element Plus 的 `<el-upload>` 组件，支持拖拽上传

**API：** `POST /api/products/:id/images`（单张）、`POST /api/products/:id/images/batch`（批量）

### 6.3 图片排序

**功能点：**

- 每张图片有 `sortOrder` 字段控制展示顺序
- 前端使用 `vuedraggable` 组件实现拖拽排序
- 排序变化时自动调用后端接口持久化
- 排序后第一张图片自动设为主图

**API：** `PUT /api/products/:id/images/reorder`

```json
// Request
{ "ids": [3, 1, 5, 2, 4] }  // 按新的顺序排列的 imageId 数组
```

### 6.4 图片删除

**功能点：**

- 删除图片记录的同时从磁盘删除文件
- 删除后自动重排剩余图片顺序
- 如果删除的是主图，将下一张设为主图

**API：** `DELETE /api/products/images/:imageId`

### 6.5 图片访问

**功能点：**

- 所有图片通过 `/api/files/**` 公开访问（无需认证）
- 设置 `Cache-Control: public, max-age=86400`（浏览器缓存 24 小时）
- 路径穿越防护：验证实际路径在存储根目录内

### 6.6 图片导出

**功能点：**

- **单张下载：** 每张图片旁有"下载"按钮，点击直接下载
- **批量导出：** 图片卡片头部"导出图片"下拉菜单，支持：
  - 全部导出：将该商品的所有图片打包为 ZIP 下载
  - 导出选中：仅导出勾选了"用于 AI"的图片
- 需要商品所有者权限

**API：** `GET /api/products/images/:imageId/export`（单张）、`POST /api/products/:id/images/export-batch`（批量 ZIP）

## 7. AI 商品描述生成

### 7.1 功能概述

利用多模态 AI 模型（通义千问 Qwen3-VL-Plus），基于商品图片和关键词自动生成结构化的商品描述。生成内容包含短标题、卖点列表、详情描述、分类、材质、颜色、风格标签、目标受众、使用场景和合规性警告。

### 7.2 触发方式

1. **单独触发：** 在商品编辑页右侧"AI 商品描述生成"面板，选择图片后点击"生成描述"
2. **流水线触发：** 在一键组合生成中勾选"描述"步骤

### 7.3 输入参数

| 参数 | 说明 | 必填 |
|------|------|------|
| 商品图片 | 可选择特定图片，不选则使用全部图片 | 否 |
| 关键词 | 用户手动输入的关键词，辅助 AI 理解 | 否 |
| 商品名称 | 自动从商品信息中获取 | 自动 |
| 已有图片数量 | 自动统计 | 自动 |

### 7.4 生成流程

```
选择商品图片 → 转 Base64 → 构建 Prompt → 调用 AI → JSON 解析 → 写入商品记录 → 扣减配额 → 记录日志
```

- 图片编码为 Base64 Data URI 嵌入请求
- AI 返回严格 JSON（`response_format: json_object`）
- JSON 解析失败时，使用备用文本模型修复 JSON
- 生成结果自动写入 `product.short_title`、`product.selling_points`、`product.detail_content`、`product.ai_analysis_raw`

### 7.5 输出内容

| 字段 | 格式 | 说明 |
|------|------|------|
| `shortTitle` | String | AI 精炼短标题，≤15 中文字符 |
| `sellingPoints` | String[] | 3-5 条核心卖点 |
| `detailContent` | String | 150-300 字详情描述 |
| `category` | String | AI 判断的商品分类 |
| `material` | String | 材质分析 |
| `color` | String | 颜色分析 |
| `styleTags` | String[] | 风格标签 |
| `targetAudience` | String | 目标受众 |
| `usageScenarios` | String[] | 使用场景 |
| `complianceWarnings` | String[] | 合规性警告（广告法） |

### 7.6 合规性检查

Prompt 中内置广告法合规要求：
- 禁止使用绝对化用语（"最好""第一""国家级"等）
- 禁止虚假医疗功效宣传
- 禁止误导性价格表述
- 不确定内容标记在 `complianceWarnings` 中

### 7.7 AI 配置

- **主力模型：** 通义千问 Qwen3-VL-Plus（DashScope）
- **备用模型：** DeepSeek Chat（文本 JSON 修复）
- **超时时间：** 连接 15s，读取 120s
- **路由配置：** `mall.ai.routing.product-description = dashscope`
- **备用路由：** `mall.ai.routing.fallback-text = deepseek`

**API：** `POST /api/ai/products/:productId/description`

## 8. AI 多平台营销文案生成

### 8.1 功能概述

基于已生成的商品描述（短标题 + 详情），为不同电商/社交平台生成定制化的营销文案。每个平台可生成多个变体版本。

### 8.2 支持平台

| 平台 | 标识 | 风格特点 |
|------|------|---------|
| **小红书** | `XIAOHONGSHU` | 活泼种草风，emoji 丰富（2-5个/段），短段落，5-10 个中文话题标签，真实体验感 |
| **淘宝** | `TAOBAO` | 专业可信赖，突出性价比和促销，清晰分点卖点，不可虚假降价 |
| **抖音** | `DOUYIN` | 口语化、短促有力适合口播，前 3 秒必须有钩子，清晰 CTA（点击/下单/领券） |
| **其它** | `OTHER` | 通用专业电商风格，结构清晰（引入→卖点→总结），语言流畅自然，适合多平台分发 |

### 8.3 触发方式

1. **单独触发：** 在商品编辑页右侧"多平台营销文案"面板，选择平台 → 点击"生成文案"
2. **流水线触发：** 在一键组合生成中勾选"营销文案"并选择平台
3. **"换一换"：** 对已有文案不满意，可点击"换一换"重新生成

### 8.4 生成参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `platform` | 目标平台 | XIAOHONGSHU |
| `variantCount` | 生成变体数量（1-3） | 3 |

### 8.5 生成流程

```
前提检查（是否有描述）→ 选取 AI Provider → 构建平台化 Prompt → 调用 AI → JSON 解析 → 
批量保存 MarketingCopy 记录 → 扣减配额 → 记录日志
```

**前提条件：** 商品的 `shortTitle` 和 `detailContent` 不能为空（需要先完成 AI 描述生成）

**API：** `POST /api/ai/products/:productId/marketing-copy`

### 8.6 输出内容

每条生成的文案包含：

| 字段 | 说明 |
|------|------|
| `platform` | 平台标识 |
| `variantNo` | 变体编号（1、2、3） |
| `title` | 文案标题 |
| `content` | 文案正文 |
| `hashtags` | 话题标签数组（如 `["#好物推荐", "#数码好物"]`） |

## 9. AI 商品详情图生成

### 9.1 功能概述

使用 DashScope Wan 2.7 图像生成模型，根据商品信息和参考图片生成电商详情页的模块图（例如产品展示板块、卖点说明板块）。生成的图片保持与原始商品外观一致。

### 9.2 触发方式

1. **单独触发：** 在商品编辑页右侧"AI 详情图生成"面板，填写参数后点击"生成详情图"
2. **流水线触发：** 在一键组合生成中勾选"详情图"

### 9.3 输入参数

| 参数 | 说明 | 必填 |
|------|------|------|
| 商品图片（参考图） | 自动使用全部商品图片作为视觉参考 | 自动 |
| `sectionTitle` | 模块标题（如"产品展示"） | 是 |
| `sectionCopy` | 文案要点（如"高品质材质，舒适耐用"） | 是 |
| `visualDirection` | 画面描述（如"简约白色背景，产品居中展示"） | 是 |
| `aspectRatio` | 宽高比 | 是 |

**支持的宽高比：**

| 选项 | 比例 | Wan 2.7 对应尺寸 |
|------|------|-----------------|
| 竖版 (3:4) | 3:4 | 2K |
| 方形 (1:1) | 1:1 | 2K |
| 横版 (16:9) | 16:9 | 2K |
| 全屏 (9:16) | 9:16 | 2K |

### 9.4 生成流程

```
上传参考图 → 构建 Wan 2.7 Prompt → 调用图像生成 API → 下载生成图片 → 
保存到本地存储 → 创建 ProductImage 记录 → 扣减配额 → 记录日志
```

- 所有商品图片作为参考图传入（Base64 编码），要求模型保持一致
- 调用 DashScope 多模态生成 API（非 OpenAI 兼容接口）
- 生成后从 OSS URL 下载图片存入本地
- 图片作为商品的新图片（`sortOrder=999`，然后调整到末尾）

**API：** `POST /api/ai/products/:productId/detail-image`

### 9.5 图片质量要求（Prompt 约束）

- 高端电商视觉风格
- 手机端适配
- 中文文字清晰可读
- 物理真实感，非网页截图

## 10. 一键组合生成（流水线）

### 10.1 功能概述

将描述生成、营销文案生成、详情图生成串联为一条自动流水线，一次触发顺序执行多步。这是系统的核心效率功能。

### 10.2 触发方式

在商品编辑页右侧"一键组合生成"面板，勾选需要的步骤并点击"一键生成"。

### 10.3 可用步骤

| 步骤 | 标识 | 前置依赖 |
|------|------|---------|
| ① AI 描述 | `description` | 无 |
| ② 营销文案 | `copy` | 描述已完成（有 `shortTitle` 和 `detailContent`） |
| ③ 详情图 | `image` | 描述已完成 |

### 10.4 执行流程

```
Step 1: 描述生成
   ↓
Step 2: 营销文案生成（使用 Step 1 的短标题和详情）
   ↓
Step 3: 详情图生成（使用 Step 1 的商品信息）
```

- 每步完成后，前端实时更新页面（描述结果自动写入表单，文案列表自动刷新，生成的图片自动出现）
- 前端使用 Element Plus 的 Steps 组件展示进度
- 如果某步失败，后续步骤仍会继续（不中断流水线）

**API：** `POST /api/ai/products/:productId/pipeline`

```json
// Request
{
  "steps": ["description", "copy", "image"],
  "platform": "XIAOHONGSHU",
  "variantCount": 3,
  "imageParams": {
    "sectionTitle": "产品展示",
    "sectionCopy": "高品质材质",
    "visualDirection": "简约白色背景",
    "aspectRatio": "3:4"
  }
}
```

**路由：** `/admin/users`

### 13.2 用户列表

**功能点：**

**功能点：**

- 新注册用户状态为 PENDING（待审核）

- 审核后状态变为 ACTIVE，记录审核人 ID 和审核时间

### 13.5 用户状态流转

```

                                        ↑                              │
                                        └──── (管理员启用) ──────────────┘
```

## 14. 配额管理（管理员）

### 14.1 功能概述

控制每个用户的 AI 生成次数。配额系统配合权限管理，实现精细化的 AI 使用控制。

### 14.2 配额字段

| 字段 | 说明 |
|------|------|
| `quotaTotal` | 总配额数。`-1` 表示不限配额 |
| `quotaUsed` | 已使用次数。每次 AI 生成成功后 +1 |

### 14.3 配额扣减

每次成功调用 AI 生成（描述、文案、图片）后：
1. 检查 `quotaTotal` 是否为 -1（不限），如果是则跳过
2. 否则检查 `quotaUsed < quotaTotal`，配额用完则拒绝
3. 生成成功后 `quotaUsed += 1`

### 14.4 设置配额

**功能点：**

- 管理员在用户管理页点击"配额"按钮
- 弹窗输入新的总配额数（输入 -1 表示不限制）
- 系统自动计算变动量（delta），写入 `user_quota_log` 审计记录

**API：** `POST /api/admin/users/:id/quota`（绝对设置）、`POST /api/admin/users/:id/quota/add`（增量追加）

### 14.5 配额审计日志

每次配额变更记录：
- 被变更用户 ID
- 操作管理员 ID
- 变动量（正数=增加，负数=扣减）
- 变动原因
- 变更时间

**数据表：** `user_quota_log`

## 15. 权限管理（管理员）

### 15.1 功能概述

在配额之外，管理员可为每个用户单独设置可用模块权限。权限控制到具体 AI 功能模块级别。

### 15.2 权限模块

| 权限标识 | 对应功能 |
|---------|---------|
| `description` | 商品描述生成 |
| `copy` | 营销文案生成 |
| `image` | 详情图生成 |

### 15.3 权限逻辑

- `permissions = null`：拥有全部权限（默认）
- `permissions = "description,copy"`：仅拥有指定模块权限
- `permissions = ""`（空字符串）：无任何模块权限
- **ADMIN 角色始终拥有全部权限**（不受此字段限制）

### 15.4 设置权限

**功能点：**

- 管理员在用户管理页点击"权限"按钮
- 弹窗显示三个复选框，勾选即授权，不勾选即限制

**API：** `POST /api/admin/users/:id/permissions`

```json
// Request
{ "permissions": "description,copy,image" }
```

### 15.5 权限检查流程

```
用户请求 AI 生成
  → 检查 isAdmin？ → 是 → 通过
  → 否 → 检查 status 为 ACTIVE？ → 否 → 拒绝
  → 检查 quotaUsed < quotaTotal（或 quotaTotal = -1）？ → 否 → 拒绝
  → 检查 permissions 是否为空？ → 是 → 通过
  → 检查 目标模块 是否在 permissions 列表中？ → 否 → 拒绝
  → 通过
```

## 22. 数据库表结构

### 21.1 表一览

| 表名 | 说明 | 关键字段 |
|------|------|---------|
| `sys_user` | 系统用户 | username, password_hash, role, status, quota_total, quota_used, permissions |
| `product` | 商品 | name, price, category_id, stock, status, short_title, selling_points, detail_content |
| `product_image` | 商品图片 | product_id, file_path, file_name, mime_type, file_size, sort_order, is_main |
| `product_category` | 商品分类 | name, parent_id, sort_order |
| `marketing_copy` | 营销文案 | product_id, platform, variant_no, title, content, hashtags, is_favorite, is_draft |
| `copy_library` | 文案库 | user_id, title, platform, content, tags |
| `ai_generation_log` | AI 生成日志 | user_id, product_id, task_type, platform, model, status, error_message, duration_ms |
| `user_quota_log` | 配额变动日志 | user_id, change_by, delta, reason |
| `system_announcement` | 系统公告 | content, created_by, is_active |

所有表使用 `utf8mb4_unicode_ci` 字符集。业务表（user、product、product_image、product_category、marketing_copy、copy_library）支持逻辑删除（`deleted` 字段）。

### 21.2 ER 关系

```
sys_user 1──N product           （用户创建商品）
sys_user 1──N marketing_copy    （用户生成文案）

product 1──N product_image      （商品图片）
```
